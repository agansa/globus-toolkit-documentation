Passing Schedular Extensions in RSL (GT 3.0)

Since the GRAM job schema (gram_rsl.xsd) supports "any elements", it is a simple matter of providing schedular-specific extension attributes to an RSL. As long as the extension attributes use any of the standard complexTypes specified in base RSL schema (rsl.xsd), the attributes will be understood and passed on to the back-end scripts. For example, adding the following to the end of a GRAM job definition would be accepted by the Managed Job service: 


<pbs:emailAddress>
    <rsl:string>
        <rsl:stringElement value="lane@mcs.anl.gov"/>
    </rsl:string>
</pbs:emailAddress>
<pbs:emailOnExecution>
    <rsl:string>
        <rsl:stringElement value="yes"/>
    </rsl:string>
<pbs:emailOnExecution>
The back-end perl scripts would then be able to use these new values

CVS

To check out code from our CVS:

  1. To login, use the following command: 
    cvs -d :pserver:anonymous@cvs.globus.org:/home/globdev/CVS/gridservices \
login
  2. At the password prompt, hit enter.

  3. Then check out code from the main trunk: 
    cvs -d :pserver:anonymous@cvs.globus.org:/home/globdev/CVS/gridservices \
co <module>
Replace with one of the following modules:
  • ogsa
  • program_execution
  • data_management
  • information_services

To check out all available modules replace <module> with "all".

Available CVS tags are:

  • globus_3_0_beta
  • globus_3_0_branch
