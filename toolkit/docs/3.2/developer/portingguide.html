Differences between 3.0.1 and 3.2

[in laymen's terms describe what changed overall?? ]

New

The following are new to Globus Toolkit 3.2:

[how do these changes affect users?]

  • XIO (which includes drivers for TCP, UDP, GSI, file, HTTP, FTP_CMD, and GSIFTP_CMD)
  • GridFTP server library
  • GridFTP server
  • IO compatibility layer

Major changes/fixes

The following are major changes and fixes between GT 3.0 and GT 3.2:

  • Core and Common
  • Common
  • GRAM Protocol IO
  • GSI
  • GridFTP
  • Index Services

Core/Common

[how do these changes affect users?]

  • Fixed problems lingering from the core/common reconfiguration from pre-GT 2.4.

Common

[how do these changes affect users?]

  • Added recursive Mutex type
  • Modified the block code to work better with callback spaces
  • Added IPv6 abstractions (compatibility layer still needed)
  • Completely re-wrote hashtable code for optimization and traversal support
  • globus_memory now uses Malloc/Free for every push/pop when [built debug]. This should aid in leak detection
  • Added a panic handler which should be used instead of assert. We will allow users to register callbacks to receive on panic.

GRAM Protocol IO

[how do these changes affect users?]

  • Fixed accesses to Globus IO handle members using new, unsupported Globus IO API Nexus.
  • Fixed accesses to Globus IO handle members and stripped UDP support.

GSI

[how does this change affect users?]

  • Use real types in GSSAPI instead of void * to aid in compile time error-checking.

GridFTP

[how do these changes affect users?]

  • Fixed the following bugs:
    • bug 1306: globus-url-copy no longer results in a "reached end of file" error after switching on logging thru the -Z flag of in.ftpd.
    • bug 1309: GridFTP can now achieve multiple (client to server) file transfer without closing the client.
    • bug 1365: >4GB files will no longer fail when the '-p N' option is used.
  • Fixed problem where setting data channel protection mode to PRIVATE used SAFE instead.

    Note: this update will likely break third party tranfers between this and any older version if data channel protection mode PRIVATE is used. So, private data channel mode is not available on any server before 1.12. When it is set to PRIVATE, it says ok, but it is actually SAFE, and will fail when communicating with a newer server that really is running in PRIVATE.
