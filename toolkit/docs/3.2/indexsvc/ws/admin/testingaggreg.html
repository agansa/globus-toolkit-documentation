Testing Service Data Aggregation

Use this simple test for notification caching, otherwise known as Service Data Aggregation. The Index Service is a persistent service, which means that to add another instance you must add an additional service descriptor entry to the Server Configuration file.

1

Open the  server-config.wsdd file in your GT3 root directory.
2

Make a copy of the existing (default) Index Service deployment descriptor.
3

Change the service name parameter to something other than Index Service.

Also, change the parameter name to change the display name of the service.

Your result should look like something like this:

<service name="base/index/IndexService2" provider="Handler" style="wrapped">     
      <parameter name="name" value="Index Service 2"/>;
     ... (no additional changes needed)</service>
4

Launch the standalone OGSA Service Container by running:

ant startContainer
5

Launch the Service Browser GUI by running:

ant gui
6

Use the Service Browser GUI to activate both Index Service instances.
7

In the Manage Subscriptions group box of the secondary Index Service instance, create a subscription to the Service Data Element (SDE) on the primary Index Service instance:

Source Enter the URL of the primary Index Service instance.
New Service Data Name Enter the ServiceData QName of the SDE on the instance to which you are subscribing. (?)
New Service Data Namespace
Sink Enter the URL of the secondary Index Service instance.
Sink Lifetime Enter a positive lifetime in seconds for the new sink.

If all goes well, you should see a message box indicating that the subscription was successful, and you should see subsequent notifications.
8

Run a query by name on the second instance of the SDE for which you created the subscription.

If a notification has occurred from the first service instance to the second, the SDE from the first instance should be returned by the second instance, where it is being cached in the services' local Service Data Container.
