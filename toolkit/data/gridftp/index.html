GridFTP is a protocol defined by Global Grid Forum Recommendation GFD.020, RFC 959, RFC 2228, RFC 2389, and a draft before the IETF FTP working group. The GridFTP protocol provides for the secure, robust, fast and efficient transfer of (especially bulk) data. The Globus Toolkit provides the most commonly used implementation of that protocol, though others do exist (primarily tied to proprietary internal systems).

GridFTP provides

  • a server implementation
  • a scriptable command line client
  • a set of development libraries for custom clients.

GridFTP in GT4

GridFTP in GT4 includes a new server implementation and the following features:

  • 100% Globus code (no licensing issues)
  • 100% backwards protocol compatible (w/ 2.4 and higher)
  • New Features:
    • Striping support is provided in 4.0
    • Has IPV6 support included (EPRT, EPSV), but we have limited environment for testing.
  • Extremely modular to allow integration with a variety of data sources (files, mass stores, etc.)
  • wuftpd specific functionality, such as virtual domains, is NOT present

GridFTP in GT3

Note: GT 3.0 is no longer supported.

While GridFTP in 3.0.x was identical feature wise to 2.4, GT 3.2 added the following features:

  • Server Improvements
    • Structured File Info
      • MLST, MLSD
    • checksum support
    • chmod support
  • •globus-url-copy changes
    • File globbing support
    • Recursive dir moves
    • RFC 1738 support
    • Control of restart
    • Control of DC security

GridFTP in GT2

Note: GT 2.x is no longer supported.

GT 2.4 introduced these features:

  • Security
  • Reliability / Restart
  • Parallel Streams
  • Third Party Transfers
  • Manual TCP Buffer Size
  • Server Side Processing (ESTO/ERET)
  • Partial File Transfer
  • Large File Support (>2GB)
  • Data Channel Caching
  • Restart/Performance

Important: GT 2.0/2.2 are deprecated and unsupported, have server stability issues AND a protocol incompatibility with later versions due to a security bug.
